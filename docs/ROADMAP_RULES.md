# Roadmap Rules

Codex may update PRODUCT_ROADMAP.md based on user feedback.

---

## 🎯 Core Principle

Only real usage feedback should influence roadmap priorities.

---

## 🔥 Priority Rules

### High Priority

- Blocks workout completion
- Causes frustration during active workout
- Causes incorrect progression behavior
- Causes data loss or inconsistency
- Repeated feedback (same issue multiple times)

### Medium Priority

- Improves speed or clarity
- Reduces friction in common flows
- Enhances understanding (progress, program, feedback)

### Low Priority

- Nice-to-have improvements
- Cosmetic changes
- Non-essential features

---

## 🚫 Do NOT Add to MVP Unless

- It directly improves the workout loop
- It is based on repeated real usage feedback
- It affects:
  - auth
  - workout logging
  - history
  - progression
  - program visibility

---

## ❌ Never Prioritize (for now)

- Social features
- Advanced analytics dashboards
- AI-generated plans
- Template builder (unless proven necessary)

---

## 🧠 Task Creation Rules

When converting feedback into tasks:

- Tasks must be concrete and actionable
- No vague tasks ("improve UX")
- Break large problems into smaller checklist items
- Prefer small improvements over big rewrites

---

## 🔄 Feedback Processing Rules

- Group similar feedback together
- Do not duplicate tasks
- Promote items only after repeated signals
- Keep roadmap focused on current priorities

---

## ⚠️ Editing Rules

- Do NOT rewrite entire roadmap
- Make minimal changes
- Preserve structure
- Add tasks only where appropriate
